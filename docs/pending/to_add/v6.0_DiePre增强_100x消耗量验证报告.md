## 三、性能瓶颈与优化建议

### 发现的问题

1. **429限速频繁** 🔴
   - GLM-5 API频繁触发429 Too Many Requests
   - 自适应限速器QPS从2.0→12.0，但仍有429
   - 已自动回退到GLM-4.7

2. **深度推演超时** 🟡
   - deep_reasoning Phase 1次超时(>150s)
   - 原因: GLM-5响应慢 + 复杂推演

3. **第1/2轮管线阻断** 🟠
   - Stage1门控未通过(sub_questions不足)
   - 可能原因: 初始节点数不足

### 优化方案

#### 短期优化 (立即可用)

1. **降低并发数**
   ```bash
   # 从8并发降至4并发
   python growth_engine.py --parallel --workers 4 --rounds 50
   ```

2. **增加重试间隔**
   ```python
   # growth_engine.py 修改AdaptiveRateLimiter
   BACKOFF_429 = 3.0  # 从2.0增加到3.0秒
   ```

3. **延长超时时间**
   ```python
   # growth_engine.py
   TIMEOUT_DEEP_REASONING = 300  # 从150s增加到300s
   ```

#### 中期优化 (需调整策略)

1. **分批执行**
   - 不要一次性并发19个Phase
   - 改为: Stage1(6并发) → Stage2(13并发) → Stage3-6(串行)

2. **智能降级**
   - 429触发3次 → 自动降低workers
   - 成功10次 → 逐步恢复workers

3. **Phase优先级**
   - 高优先级: code_domain (权重0.95)
   - 中优先级: bottom_up, deep_reasoning
   - 低优先级: horizontal (权重0.70)

#### 长期优化 (架构改进)

1. **本地模型混合**
   - GLM-5用于关键Phase (falsify, code_domain)
   - Qwen2.5-Coder:14b用于辅助Phase (horizontal)

2. **增量式成长**
   - 不是每轮都跑全部19个Phase
   - 根据收敛状态动态调整Phase组合

3. **分布式部署**
   - 多机部署，每机独立API Key
   - 负载均衡分配Phase
