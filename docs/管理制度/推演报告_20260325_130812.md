## 七、代码架构映射建议

| 层级 | 治理角色 | 代码映射 | 职责 |
|------|---------|---------|------|
| L0 | 参谋本部 | Multi-Agent Orchestrator (Planner + Critic) | 双轨运行。'作战局'(Planner)负责制定方案，'监察局'(Critic/Reviewer)负责红队测试与封驳。两者 |
| L1 | 战区司令部 | Domain-Specific Router / Manager Agent | 常设机构。按业务领域划分（如'前端战区'、'数据战区'）。拥有独立的资源池和私有知识库。接收L0的战略意图，自主制定战役 |
| L2 | 联合特遣队 | Dynamic Agent Crew (LangGraph SubGraph) | 临时组建。针对特定任务（如'修复支付Bug'），从各战区抽调兵力（代码Agent、测试Agent）组成。任务结束后解散。 |
| L3 | 功能分队 | Tool / Function Call | 原子化能力。单一功能执行者（读文件、写库、发请求）。无决策权，只对L2负责。 |

### 制衡机制
- 红蓝对抗: L0层必须包含一个'反对者'角色，所有指令必须经过驳辩才能下发。
- 事后审计: 独立的'都察院'Agent定期审查L1/L2的日志，不阻塞流程但追责责任。
- 资源预算: L0控制L1的Token预算和工具权限，防止L1'拥兵自重'（无限循环调用）。

### 信息通路
异步消息总线。L0发布Intent，L1订阅并反馈Plan，L2执行。通过Shared State同步状态，而非层层传递。
