## 三、MXFP4 MoE (amd-moe-mxfp4)

### 版本历史
| 版本 | 关键变更 | 结果 |
|------|---------|------|
| v1 baseline | 直接调用fused_moe (与reference相同) | ~185µs |
| v3 | CUDA Graph capture fused_moe | benchmark ~1µs, **leaderboard无效** |
| v5 | Shared expert分离 (dense GEMM) | **不划算, 放弃** |
| v7 | 参数调优 + pad缓存 | 未测试 |
| v8 | doweight_stage1=True | ❌ **全部正确性测试失败** |
| v10 (当前) | 修复: doweight_stage1=False + expert_mask | 待提交 |

### v8 失败根因分析
**doweight_stage1=True 导致正确性失败**:
- 数学上: `w_i * SiLU(gate) * up` → `down` ≡ `SiLU(gate) * up` → `w_i * down` (线性等价)
- 实际: MXFP4量化在Stage1→Stage2之间执行
- `quant(w_i * intermediate) ≠ w_i * quant(intermediate)` (量化误差非线性)
- 结果: 4077~2105043个元素不匹配 (所有测试case全部失败)

### v10 优化策略
1. **修复正确性**: doweight_stage1=False (匹配reference量化路径)
2. **expert_mask**: 小bs+大E时跳过空expert
   - 条件: `sparsity < 4.0 and n_experts > 64`
   - bs=16, E=257, top_k=9 → sparsity=0.56 → 启用mask (仅~16个expert活跃)
   - bs=512, E=257, top_k=9 → sparsity=17.9 → 不启用 (overhead > 收益)
3. **pad缓存**: 避免每次重新计算padding差值

## 四、关键教训

1. **benchmark ≠ leaderboard**: benchmark模式data不变, leaderboard每次regenerate
2. **缓存/Graph在leaderboard无效**: PyTorch CUDA allocator复用地址导致cache miss → 错误结果
3. **doweight_stage1与量化不兼容**: CK kernel的inter-stage量化使stage1 weighting不等价
4. **config-keyed缓存是安全的**: metadata只依赖(bs,qsl,kvsl), 不依赖seed
5. **几何均值排名**: 最慢case影响最大, 优化应优先处理最大case
