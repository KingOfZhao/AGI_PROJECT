# 系统思维框架

> 来源: 训练知识结构化提取 | 用于: 根因分析、架构决策、考试reasoning题

## 0. 知识边界的认知（已知/未知框架）

### 已知(F)和未知(V)不仅适用于技术问题
```
技术已知: 公式、数据、代码——可验证
技术未知: 未探索的技术栈——可规划路径
认知已知: 我知道自己知道的——可结构化
认知未知: 我不知道自己不知道的——需要"虚"来探索
```

### "虚"是探索认知未知的核心工具
- 非虚构传递已知的已知（F→F）
- 小说/叙事探索未知的已知和未知的未知（V→F）
- 人类工匠的经验（手感、直觉）介于F和V之间，需要叙事来捕捉
- DiePre项目中工匠"知道但说不出来"的知识 = V层，需要叙事性对话转化为F

### 认知偏误警告
- 评估知识价值时，倾向于高估可量化的（F），低估不可量化的（V）
- "小说性价比低"是一种已知偏误——用F的评估标准去衡量V的价值
- 正确做法：对V类知识用不同的评估维度（扩展边界、同理心、灰色地带理解）

## 1. 反馈回路识别

### 正反馈（增强回路）
- 定义: 输出放大输入，导致指数增长或崩溃
- 模式: A↑ → B↑ → A↑↑
- 例子: 缓存穿透 → DB负载↑ → 响应变慢 → 更多超时 → 重试风暴 → DB负载↑↑↑
- 关键信号: "恶性循环"、"滚雪球"、"雪崩"

### 负反馈（平衡回路）
- 定义: 输出抑制输入，维持稳定
- 模式: A↑ → B↓ → A↓
- 例子: 负载↑ → 自动扩容 → 容量↑ → 负载↓
- 关键信号: "自愈"、"稳定"、"回归"

### 识别方法
1. 画出变量之间的因果箭头
2. 追踪闭合回路
3. 判断每个回路是增强还是平衡
4. 找主导回路（时间尺度决定哪个回路主导）

## 2. 系统杠杆点（Meadows 12个，从弱到强）

| 级别 | 杠杆点 | 示例 |
|------|--------|------|
| 12 | 常量/参数/数字 | 调超时时间、改连接池大小 |
| 11 | 缓冲区大小 | 增加队列容量 |
| 10 | 物质流/存量结构 | 改缓存层级 |
| 9 | 传输延迟 | 调整TTL、重试间隔 |
| 8 | 正反馈回路强度 | 关掉级联重试 |
| 7 | 负反馈回路强度 | 改进告警阈值 |
| 6 | 信息流透明度 | 让开发者看到系统状态 |
| 5 | 系统规则 | 改部署流程（加review） |
| 4 | 自组织能力 | 允许服务自主降级 |
| 3 | 系统目标 | 从"功能交付"改为"系统稳定性" |
| 2 | 系统范式 | 从"集中式"改为"分布式" |
| 1 | 超越范式的力量 | 重新定义问题本身 |

**考试应用**: "如何防止生产事故"→ 不要只答"加监控"(级别12)，而是"建立blameless文化"(级别5) + "允许服务降级"(级别4)

## 3. 根因分析框架

### 5 Whys
```
问题: API响应时间p99=30s
1. 为什么慢? → DB连接池耗尽
2. 为什么耗尽? → 慢查询占用连接
3. 为什么慢查询? → 缺少索引
4. 为什么缺少索引? → 没有索引审查流程
5. 为什么没有流程? → 没有把DB schema变更纳入review
→ 根因: 流程缺失，不是技术bug
```

### 因果链重建（考试常用）
1. 收集所有artifacts（日志、时间戳、IP）
2. 按时间排序
3. 识别初始访问向量
4. 追踪攻击/故障链
5. 确定最终影响

### 瑞士奶酪模型
```
防线1[漏洞] → 防线2[漏洞] → 防线3[漏洞] → 事故
没有单点故障，是多道防线同时失效
→ 修复策略: 不只补一层，要增加纵深
```

## 4. 分布式系统核心原则

### CAP定理
- 一致性(C) + 可用性(A) + 分区容忍(P): 只能同时满足2个
- 实际选择: CP（银行）或 AP（社交媒体）
- **关键洞察**: 网络分区一定会发生，所以实际是在C和A之间选

### 分布式系统八大谬误
1. 网络是可靠的
2. 延迟为零
3. 带宽无限
4. 拓扑不会变
5. 只有一个管理员
6. 传输成本为零
7. 网络同构
8. 系统是安全的

### 故障模式
| 类型 | 表现 | 应对 |
|------|------|------|
| 崩溃故障 | 进程停止 | 重启 + 健康检查 |
| 遗漏故障 | 请求/响应丢失 | 超时 + 重试 + 幂等 |
| 时序故障 | 响应延迟 | 断路器 + 降级 |
| 拜占庭故障 | 任意错误行为 | 共识协议（Paxos/Raft） |

## 5. SRE核心概念

### SLI/SLO/SLA层次
- **SLI**: 服务质量指标（如p99延迟、错误率）
- **SLO**: 目标值（如p99<300ms）
- **SLA**: 与客户的合同承诺（违反有赔偿）
- **Error Budget**: 100% - SLO = 允许的不可用时间

### 错误预算策略
```
SLO = 99.9% → 月度允许停机 = 43.2分钟
已用 = 40分钟 → 剩余3.2分钟 → 停止新功能发布，专注稳定性
已用 = 5分钟 → 剩余38.2分钟 → 可以承受风险，继续发布
```

### BLAST原则（事故处理）
- **B**elieve the user's report（相信用户报告）
- **L**earn from all failures（从所有故障中学习）
- **A**nticipate failures（预见故障）
- **S**implify the solution（简化解决方案）
- **T**reat operations as a first-class citizen（把运维当一等公民）
