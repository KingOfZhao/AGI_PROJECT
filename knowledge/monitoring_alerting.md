# 监控与告警设计模式

> 来源: 训练知识结构化提取 + DDIA章节笔记 | 用于: 考试retrieval题、生产系统设计

## 1. 告警设计原则

### 好告警 vs 坏告警
| 好告警 | 坏告警 |
|--------|--------|
| 可行动（你知道该做什么） | 信息性（只告诉你"出事了"） |
| 关联runbook | 没有runbook |
| 有阈值分级（warn/critical） | 只有一个级别 |
| 低误报率（<5%） | 频繁误报导致"狼来了" |
| 指向具体系统组件 | 模糊的"服务不可用" |

### 告警疲劳管理
- 每人每月被pager打扰 < 10次
- 超过阈值→需要优化告警或修复系统
- 分级通知: Critical=PagerDuty, Warning=Slack, Info=Dashboard

## 2. 关键指标设计

### RED指标（服务端）
- **R**ate: 每秒请求数
- **E**rrors: 错误率（4xx/5xx占比）
- **D**uration: 延迟分布（p50/p95/p99）

### USE指标（资源端）
- **U**tilization: 资源使用率
- **S**aturation: 排队/等待深度
- **E**rrors: 错误数

### 四个黄金信号（Google SRE）
1. 延迟: 请求处理时间分布
2. 流量: 系统负载（QPS）
3. 错误: 请求失败率
4. 饱和度: 资源使用程度

## 3. 告警阈值设计

### 静态阈值 vs 动态阈值
```
静态: p99 > 300ms → alert
优点: 简单、可预测
缺点: 无法适应正常波动（如早晚高峰）

动态: p99 > 历史基线 + 3σ → alert
优点: 适应周期性变化
缺点: 复杂，需要足够历史数据
```

### 多级阈值示例
```
服务可用性:
  Warning:  连续2次健康检查失败（30s）
  Critical: 连续5次失败 或 持续失败 > 2min

延迟:
  Warning:  p99 > SLA的80%
  Critical: p99 > SLA的100%

错误率:
  Warning:  > SLO的50%
  Critical: > SLO的100%
```

## 4. 分布式追踪设计

### Trace → Span → Log 三层
```
Trace ID: 全局唯一，贯穿整个请求链路
  ├─ Span 1: API Gateway (50ms)
  │   ├─ Span 1.1: Auth Service (20ms)
  │   └─ Span 1.2: User Service (30ms)
  │       └─ Span 1.2.1: DB Query (15ms)
  └─ (每条span可以关联日志)
```

### 实现要点
- Trace ID通过HTTP header传播（X-Request-ID）
- 每个服务添加span开始/结束时间
- 采样策略: 全量（低流量）或概率采样（高流量）

## 5. Runbook模板

```markdown
# [告警名称] Runbook

## 概述
[一句话描述这个告警的含义]

## 立即检查
1. [ ] Dashboard: [链接]
2. [ ] 是否有相关部署？（查看最近1小时）
3. [ ] 是否有上游依赖故障？

## 常见原因
| 原因 | 症状 | 修复 |
|------|------|------|
| DB慢查询 | p99高，DB CPU高 | kill查询，加索引 |
| 连接池耗尽 | 大量TimeoutError | 扩容或限流 |

## 临时缓解
[快速止血方法]

## 根因排查
[详细排查步骤]

## 升级路径
5分钟无进展 → 通知[人]
15分钟无进展 → 升级到[负责人]
```
