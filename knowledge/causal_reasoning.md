# 因果推理框架

> 来源: 训练知识结构化提取 | 用于: 事故分析、数据解读、考试reasoning题

## 1. 相关性 ≠ 因果性

### 虚假相关
```
现象: 部署缓存后，错误率从0.1%升到0.5%
错误推理: "缓存导致更多错误"（相关性）
正确推理: 部署缓存 → 响应变快 → 更多用户涌入（流量+40%）→ 更多请求总量 → 
          更多错误绝对数 → 但错误率上升可能是因为新流量来自异常来源
需要检查: 基线、混淆变量、时间顺序
```

### 因果推理三要素（Hill标准简化版）
1. **时间顺序**: 原因必须在结果之前
2. **剂量-反应**: 越多原因 → 越多结果
3. **混杂变量**: 是否有第三方因素同时影响两者？

## 2. 因果链重建（安全事件分析）

### 步骤
1. **列出所有artifacts**（时间、IP、操作、状态）
2. **按时间排序**（绝对时间，不是相对顺序）
3. **识别初始访问向量**（最早的可疑活动）
4. **追踪横向移动**（权限提升、持久化）
5. **确定数据影响**（什么被访问/修改/泄露）
6. **验证链完整性**（每个环节是否有证据支撑）

### 常见攻击链模式
```
初始访问 → 权限提升 → 横向移动 → 持久化 → 数据窃取 → 掩盖痕迹

初始访问常见方式:
- 钓鱼邮件 → 凭证窃取
- 公开暴露的服务（webhook/API未认证）
- 供应链攻击（依赖注入）
- SQL注入 → 数据库凭证泄露

权限提升常见方式:
- 提权漏洞
- 凭证复用
- 过度权限（deploy用户可读生产密钥）
```

## 3. 多重因果 vs 单一因果

### 线性思维（错误）
"缓存导致故障" → 一个原因，一个结果

### 系统思维（正确）
"缓存配置不当 + 无断路器 + 重试风暴 + 连接池耗尽" → 多因素叠加

### 关键问题
- 去掉任何一个因素，事故是否还会发生？
- 如果是→那个因素不是必要条件
- 如果否→那个因素是必要条件

## 4. 反事实推理

### 方法
"如果当时没有做X，Y还会发生吗？"
- 如果不会 → X可能是原因
- 如果会 → X不是原因

### 应用
```
部署新版本后系统变慢
反事实: 如果没部署，系统会变慢吗？
→ 检查: 是否有其他变化？（流量高峰、上游服务故障、磁盘满）
→ 如果其他因素也能解释 → 不能确定是新版本导致的
```

## 5. 数据解读陷阱

### 幸存者偏差
- 只看到成功案例，忽略失败案例
- 例子: "成功的企业都早起" → 忽略了大量早起但失败的企业

### 回归均值
- 极端表现倾向于回归平均
- 例子: "重构后性能提升20%" → 可能只是之前恰好性能低谷

### 停止规则偏误
- 提前停止实验会扭曲结果
- 例子: "测试5个用户，4个喜欢" → 样本太小，结论不可靠

### Simpson悖论
- 子群体趋势与整体趋势相反
- 例子: A方案在Segment1更好，Segment2也更好，但整体看起来更差
- 原因: 群体大小不均
