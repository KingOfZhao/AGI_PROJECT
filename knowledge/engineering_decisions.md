# 工程决策框架

> 来源: 训练知识结构化提取 | 用于: 技术选型、优先级排序、考试EQ/und题

## 1. 技术选型决策矩阵

### 评估维度（按权重）
1. **团队熟悉度** (30%) - 选团队能维护的，不是"最好"的
2. **运维复杂度** (25%) - TCO比首月成本重要10倍
3. **生态系统** (20%) - 社区、文档、招聘难度
4. **性能需求** (15%) - 实际测量，不是基准测试
5. **未来扩展** (10%) - YAGNI原则，但不做死胡同选择

### 常见选型反模式
- ❌ "X是大公司用的，所以适合我们"
- ❌ "Y是最新的，我们应该用"
- ✅ "在我们的具体场景下，X的Z特性解决了我们的具体问题"

## 2. 优先级决策框架

### RICE评分
```
Reach (影响用户数) × Impact (单用户影响程度 1-3) × 
Confidence (确信度 0.1-1.0) / Effort (人月)
```

### MoSCoW优先级（真实用法）
- **Must**: 不做就不发布
- **Should**: 重要但可延后一个周期
- **Could**: 有空就做
- **Won't**: 明确不做（不是"以后做"）

### 优先级冲突解决
1. 时间冲突: A必须在B之前（依赖关系）
2. 资源冲突: 哪个ROI更高？
3. 人员冲突: 谁来做？swap不好swap的？
4. 目标冲突: 业务目标 vs 技术债务

## 3. 架构决策记录（ADR）

### 模板
```markdown
# ADR-NNN: [标题]

## 背景
为什么需要做这个决策？当前状态是什么？

## 决策
我们决定 [做什么]。

## 考虑的选项
### 选项A: [名称]
- 优点: ...
- 缺点: ...
### 选项B: [名称]
- 优点: ...  
- 缺点: ...

## 理由
选择[X]因为...
权衡了...和...后...
```

### 关键原则
- 记录"为什么"，不是"是什么"
- 承认不确定性（"我们目前相信..."）
- 设定回顾日期（6-12个月后重新评估）

## 4. 代码审查框架

### 审查优先级（按风险排序）
1. 🔴 安全漏洞（SQL注入、XSS、认证绕过）
2. 🔴 数据完整性（外键、约束、迁移不可逆）
3. 🟡 错误处理（裸except、吞掉错误）
4. 🟡 性能（N+1查询、缺失索引、无分页）
5. 🟢 可读性（命名、结构、注释）
6. 🟢 风格（格式、lint）

### 审查反馈原则
- 具体可执行（"第23行缺少参数校验" vs "代码不好"）
- 区分must-fix和suggestion
- 对新手：先肯定价值，再指出问题，最后提供指导
- 不使用绝对语言（"必须"→ "建议"，"错误"→ "风险"）

### Disagree and Commit
- 讨论阶段：充分表达不同意
- 决策后：全力执行团队决定
- 保留不同意见的记录
- 事后如果决策错误，说"I told you so"不可接受

## 5. YAGNI原则应用指南

### 应该做的（YAGNI ✅）
- 当前用户需求的功能
- 系统稳定性需要的
- 合规/安全要求的

### 不应该做的（YAGNI ❌）
- "未来可能需要"的抽象
- "学新框架"的技术驱动重构
- "为了灵活"的过度配置化
- 5个用户月度功能用K8s微服务架构

### 例外
- 如果改动的成本极低（<1小时），可以适度提前
- 如果不改会造成不可逆的技术债务（如数据库schema）
- 如果有明确的时间线（"Q3一定需要"）
