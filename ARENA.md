# 伟人竞技场 — 多Agent互相验证体系

> 核心理念：产出必须经过至少2个独立验证才算完成
> 不是一个人自测自嗨，是多个视角碰撞出真实质量

## 竞技场角色

### 🎯 执行者（Executor）
- **模型**: glm-5-turbo
- **职责**: 写代码、改功能、部署
- **产出**: 代码/配置/文档

### 🔍 审查者（Reviewer）
- **模型**: glm-5
- **职责**: 审查执行者的产出，找bug、逻辑漏洞、安全隐患
- **产出**: 审查报告（pass/fail + 具体问题）

### 🧠 反思者（Reflector）
- **模型**: glm-5-turbo
- **职责**: 从失败中提炼教训，更新ERRORS.md和LEARNINGS.md
- **产出**: 经验文档更新

### 📊 对标者（Benchmark）
- **模型**: glm-4.7（低成本）
- **职责**: 定期运行BrowseComp测试，追踪能力变化
- **产出**: 成绩记录 + 趋势分析

## 流程

```
1. 执行者完成任务 → 产出代码/文档
2. 审查者独立审查 → 通过/打回（附具体问题）
3. 如果打回 → 执行者修复 → 审查者再次审查
4. 通过后 → 反思者提炼经验 → 更新ERRORS.md/LEARNINGS.md
5. 每周日 → 对标者运行BrowseComp → 更新能力分数
```

## 验证标准

| 产出类型 | 审查检查项 |
|---------|-----------|
| 代码 | import完整、无裸except、参数校验、本地测试通过 |
| API | 状态码正确、错误处理、参数校验、返回格式一致 |
| 文档 | 信息准确、可操作、无过时数据 |
| 部署 | checklist全通过、服务可用、回滚方案就绪 |

## 实现方式

通过OpenClaw subagent实现：
- 每个角色是独立的session
- 主Agent负责调度和最终决策
- 审查结果写入CRM assign_history
